# master.md — 다인아이앤씨 데이터 통합 프로젝트 목표 문서

> 이 문서는 회사의 모든 정보를 **노션(Notion)** 으로 통합하고, 이를 기반으로 **실시간 통합 대시보드**와 부가 기능을 구축하기 위한 목표·방향을 정리한 마스터 문서입니다.
> 최종 수정: 2026-06-16 · 작성자: 김동만 (팀장)

---

## 1. 한 줄 비전

> **"회사의 모든 데이터를 노션 하나로 통합하고, 어디에 있든 실시간으로 조회·활용할 수 있는 통합 대시보드를 만든다."**

- 모든 데이터의 **원본(Single Source of Truth)은 노션**이다.
- 흩어진 도구(슬랙·채널톡·구글캘린더·구글드라이브·구글시트)의 데이터를 노션으로 모으고, 노션을 **종착점**으로 삼는다.
- 통합된 데이터를 한눈에 보는 **실시간 대시보드**를 구성하고, 그 위에 업무 자동화·분석 기능을 얹는다.

**다인아이앤씨 직원 명단(4명): 김동만(팀장)·민승재·김소원·조아름.**

## 3. 현재 사용 중인 도구 (As-Is)

| 영역 | 도구 | 역할 | 노션과의 관계 |
|------|------|------|----------------|
| 사내 메신저 | **슬랙(Slack)** | 내부 커뮤니케이션 | 알림·자동화 연동 대상 |
| 고객 응대(CS) | **채널톡(Channel.io)** | 고객 문의·상담 | 상담/고객 데이터 → 노션 수집 |

| 자료 관리 | **구글 드라이브** | 파일·문서 저장 | 파일 링크·메타데이터 노션 연결 |
| 사내 데이터 | **구글 시트 / 노션** | 데이터 관리 | 시트 → 노션으로 통합 이전 |
| **통합 종착점** | **노션(Notion)** | **모든 데이터의 원본 DB** | ⭐ 최종 통합 지점 |

## 4. 목표 아키텍처 (To-Be)

```
   [슬랙]   [채널톡]   [구글캘린더]   [구글드라이브]   [구글시트]
     │         │           │              │             │
     └─────────┴───────────┴──────────────┴─────────────┘
                            │  (수집 · 동기화)
                            ▼
                  ┌───────────────────┐
                  │   노션 (원본 DB)   │  ← Single Source of Truth
                  │  - 고객 / CS DB    │
                  │  - 일정 / 업무 DB  │
                  │  - 자료 / 문서 DB  │
                  │  - 내부 운영 DB    │
                  └─────────┬─────────┘
                            │  (API 조회)
                            ▼
                  ┌───────────────────┐
                  │  통합 대시보드      │  ← 실시간 조회 · 시각화
                  │  - 경영 지표        │
                  │  - CS 현황          │
                  │  - 일정 / 할 일     │
                  │  - 부가 기능        │
                  └───────────────────┘
```

**핵심 원칙**
1. **노션이 원본** — 다른 도구는 입력 채널이거나 보조 표시 수단일 뿐, 데이터의 기준은 노션.
2. **단방향 우선** — 외부 도구 → 노션으로 모으는 흐름을 먼저 정립하고, 필요한 경우에만 양방향 동기화.
3. **실시간 조회** — 대시보드는 노션 데이터를 실시간(또는 짧은 주기)으로 읽어 표시.
4. **대시보드는 React로 직접 구축** — 서드파티 도구가 아닌 자체 React 앱으로 만들어 자유로운 UI·기능 확장을 확보. (아래 §5.5 참고)

## 5. 노션 DB 통합 설계 (초안)

> 노션 안에서 관리할 핵심 데이터베이스. 각 DB는 명확한 1차 키와 관계(Relation)를 가진다.

| DB 이름 | 주요 내용 | 데이터 출처 |
|---------|-----------|-------------|
| **고객 DB** | 고객사·담당자·연락처·등급 | 채널톡, 구글시트 |
| **CS / 상담 DB** | 문의 내역·처리 상태·담당자 | 채널톡 |
| **일정 DB** | 미팅·마감·업무 일정 | 구글캘린더 |
| **업무 / 할 일 DB** | 프로젝트·태스크·진행 상태 | 노션(직접 입력) |
| **자료 / 문서 DB** | 파일 링크·버전·소유자 | 구글드라이브 |
| **운영 / 지표 DB** | 매출·비용·KPI 등 경영 데이터 | 구글시트 |

> 🔗 DB 간 **Relation**으로 연결(예: CS DB ↔ 고객 DB, 업무 DB ↔ 일정 DB)하여 통합 조회를 가능하게 한다.

## 5.5 대시보드 기술 구조 (확정: React 기반)

> 대시보드는 서드파티 도구가 아닌 **자체 React 앱**으로 구축한다. 이유: UI 자유도, 기능 확장성, 자산화.

### 데이터 흐름

```
   [노션 DB] ──(Notion API)──> [백엔드/BFF] ──(REST/JSON)──> [React 대시보드]
                                    │                            │
                              캐싱·집계·인증              조회·시각화·인터랙션
```

> ⚠️ 노션 API 키는 **브라우저(React)에 직접 노출하면 안 됨** → 사이에 가벼운 백엔드(BFF) 계층을 두어 키 보관·캐싱·집계를 처리한다.

### 기술 스택 (제안 — 확정 전)

| 구분 | 제안 | 비고 |
|------|------|------|
| **빌드/프레임워크** | React + **Vite** (또는 Next.js) | Next.js는 백엔드(API Routes)까지 한 번에 해결 가능 |
| **언어** | **TypeScript** | 데이터 구조가 많아 타입 안정성 권장 |
| **UI 라이브러리** | Tailwind CSS + shadcn/ui (또는 MUI) | 빠른 대시보드 UI 구성 |
| **차트/시각화** | Recharts (또는 Tremor) | 지표·추이 그래프 |
| **데이터 패칭** | TanStack Query (React Query) | 캐싱·실시간 갱신·로딩 처리 |
| **백엔드(BFF)** | Next.js API Routes / Node(Express) | 노션 API 호출·키 보관·집계 |
| **노션 연동** | `@notionhq/client` (공식 SDK) | DB 조회·필터·정렬 |
| **인증** | 사내 5인용 간단 로그인(구글 OAuth 등) | 역할별 뷰 제어 |
| **배포** | Vercel (또는 사내 서버) | Next.js와 궁합 좋음 |

> 💡 **추천 조합**: 백엔드를 따로 만들 부담을 줄이려면 **Next.js + TypeScript + Tailwind/shadcn + TanStack Query + Recharts + Vercel** 조합이 5인 규모에 가장 효율적입니다. (프론트와 노션 API 백엔드를 한 프로젝트에서 처리)

### "실시간"의 현실적 구현
- 노션 API는 웹소켓 푸시가 없으므로 **폴링(주기적 조회)** 방식 사용.
- TanStack Query의 `refetchInterval`로 30초~수분 주기 갱신 → 체감 실시간.
- 백엔드에서 짧게 캐싱하여 노션 API 호출 한도(초당 ~3req)를 보호.

## 6. 통합 대시보드 목표 기능

### 6.1 1차 목표 (조회·가시성)
- [ ] 노션 데이터를 한 화면에서 실시간 조회
- [ ] 역할별 뷰: 대표(경영 지표) / 팀장(전체 현황) / 직원(내 업무)

### 6.2 2차 목표 (자동화·연동)
- [ ] 외부 도구 → 노션 자동 수집(채널톡 상담, 구글캘린더 일정 등)
- [ ] 슬랙 알림 연동(마감 임박·신규 문의·지표 이상)
- [ ] 구글시트 ↔ 노션 데이터 동기화

### 6.3 3차 목표 (분석·확장)
- [ ] 기간별 매출·CS·업무 통계 및 추이 분석
- [ ] 반복 업무 자동화 / 리포트 자동 생성
- [ ] 향후 ERP 수준의 통합 운영 체계로 확장

## 9. 원칙 (Do / Don't)

**Do**
- 노션을 항상 데이터의 **원본**으로 취급한다.
- 새 데이터/기능을 추가할 때 "이게 노션 어느 DB에 들어가는가?"를 먼저 정한다.
- 5인 규모에 맞는 **단순하고 유지보수 쉬운** 방식을 우선한다.

**Don't**
- 같은 데이터를 여러 도구에 **중복 원본**으로 두지 않는다.
- 통합 전에 복잡한 자동화부터 만들지 않는다(데이터 구조 먼저).
- 검증 없이 양방향 동기화를 도입하지 않는다(데이터 충돌 위험).

## 10. README 작성 규칙

README 파일을 작성·수정할 때는 아래 **4가지 역할**을 항상 충족하도록 한다.
(저장소 메인에 노출되는 "프로젝트의 얼굴"이므로, 코드를 보지 않아도 이해되게 쓴다.)

1. **첫인상 / 프로젝트 소개**
   - 이 프로젝트가 **뭘 하는지, 왜 만들었는지**를 방문자가 코드를 안 봐도 바로 알 수 있게 설명한다.
   - 처음 온 사람(미래의 나 포함)이 "이게 뭐지?"를 해결해주는 문서다.

2. **사용 설명서**
   - 설치 방법, 실행 방법, 필요한 환경(예: Node 버전, `.env` 설정)을 적어 **다른 사람·다른 PC에서도 똑같이 돌릴 수 있게** 한다.
   - 서버 배포 같은 작업도 적어두면 나중에 그대로 따라 하기만 하면 된다.

3. **자동으로 메인에 표시됨**
   - 저장소 **루트(최상위 폴더)** 에 `README.md` 를 둔다 → GitHub가 메인 화면에 렌더링한다.
   - **마크다운**으로 작성한다(제목·표·코드블록·이미지 적용).

4. **협업·유지보수의 기준점**
   - 팀원이 프로젝트를 이해하고, **어떻게 기여하는지, 폴더 구조가 뭔지** 파악하는 출발점이 되게 한다.

5. **버전 변경 이력 (Changelog)**
   - **직전 버전과 비교해 무엇이 달라졌는지**(추가 · 수정 · 삭제)와 **변경된 주요 파일**을 기록한다.
   - 방문자·팀원이 버전 간 차이와 진행 상황을 한눈에 파악할 수 있게 한다.
   - 형식 예: `## 변경 이력` 섹션에 버전별로 **날짜 · 요약 · 주요 변경 파일**을 정리.

> 이 규칙은 별도 지시가 없어도 README 작성 시 기본으로 적용한다.

## 11. 대화 세션 관리 규칙 (Claude Code)

**작업이 끝나거나 주제가 바뀌면, Claude 가 먼저 "여기서 마무리하고 `/clear` 하시죠"라고 제안한다.** 사용자가 먼저 묻기를 기다리지 않는다.

### 왜 필요한가

Claude 는 질문할 때마다 그 대화 전체를 처음부터 다시 읽는다. 그래서 **대화가 길어지면 질문 한 번의 비용이 그대로 커진다.**

2026-07-22 실측:

| | 평균 컨텍스트 | 질문 수 | 하루 소비 |
|---|---|---|---|
| 실제 | 612k | 294회 | **180.3M** |
| 200k로 유지했다면 | 200k | 294회 | 59M (**1/3**) |

같은 일을 하고도 3배를 쓴 셈이다. 앞서 6/29~7/22 한 달간 세션 하나를 안 끊고 쓴 결과, 전체 토큰의 **97.7%가 캐시 재읽기**로 나갔고 7/20에는 구독 한도가 소진돼 API 과금으로 넘어갔다.

### 언제 제안하는가

아래에 해당하면 작업을 마무리하고 `/clear` 를 권한다.

- 요청한 작업이 **완료·검증·커밋까지 끝났을 때**
- 사용자가 **다른 주제를 꺼냈을 때** (예: 블로그 작업 → 당근 이야기)
- 조사·디버깅이 끝나고 **결론이 나왔을 때** (긴 로그·덤프가 컨텍스트에 쌓여 있다)
- 컨텍스트 경고 훅이 뜬 뒤 **하던 작업이 일단락됐을 때**

반대로 **작업 중간에는 권하지 않는다.** 맥락이 끊기면 오히려 손해다.

### 어떻게 제안하는가

- 답변 끝에 한두 줄로 짧게. 길게 설명하지 않는다.
- 무엇이 끝났고 다음에 뭘 하면 되는지 함께 적어, 새 대화에서 바로 이어갈 수 있게 한다.
- 기록은 자동 저장되므로 **잃는 것이 없다**는 점을 알린다.

### 사용자가 알아둘 것

- `/clear` 는 삭제가 아니다. 대화는 `~/.claude/projects/` 에 남고, 매일 00:05 에 `D:\erp-chatlogs\채팅_YYYY-MM-DD.md` 로도 백업된다.
- 예전 대화를 **읽기만** 할 거면 `/resume` → `Space` 미리보기. 열어서 이어가면 무거운 컨텍스트가 통째로 되살아난다.
- 작업이 아직 안 끝났는데 길어지면 `/clear` 대신 `/compact`.
- 컨텍스트 300k · 500k · 700k 에서 자동 경고가 뜬다 (`.claude/hooks/context-warn.mjs`).

> 이 규칙은 별도 지시가 없어도 모든 세션에서 기본으로 적용한다.
